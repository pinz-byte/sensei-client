# M2 — Independent Consumer Station

**Role:** Pulls `sensei_client` as a dependency. Runs ventures/projects that integrate with SENSEI. Does not author the library; does not run the service.

**Network:** Must reach M3 on port 8000.

## Bootstrap

M2 does not start with the sensei-client repo. First clone it, then bootstrap.

```bash
# 1. Clone the public repo (no auth required — it's public)
git clone https://github.com/pinz-byte/sensei-client.git ~/sensei-client
cd ~/sensei-client

# 2. Bootstrap M2 role
./setup/bootstrap.sh M2 --m3-url http://<M3_HOST>:8000
```

What it does on M2 specifically:

1. Verifies Python ≥ 3.9
2. Creates `venv/` in the clone root
3. Installs `sensei_client` from `git+https://github.com/pinz-byte/sensei-client.git@v0.3.2` — pinned to the tag, not `main`
4. Writes `.env` with `SENSEI_STATION=M2` and the M3 URL
5. Runs the smoke test against M3

## Typical M2 usage — inside a venture project

Bootstrap leaves M2 with `sensei_client` installed in `~/sensei-client/venv`. To use it in a separate project:

```bash
# In your project directory
python3 -m venv venv
source venv/bin/activate
pip install "git+https://github.com/pinz-byte/sensei-client.git@v0.3.2"

# Load env vars from the sensei-client bootstrap
set -a; source ~/sensei-client/.env; set +a
# → SENSEI_API_URL is now set; your project can consume it

# Run your project
python main.py
```

Alternative: point your project's pyproject.toml at the pinned tag:

```toml
dependencies = [
    "sensei_client @ git+https://github.com/pinz-byte/sensei-client.git@v0.3.2",
]
```

Then `pip install -e .` in the project and sensei_client comes in transitively.

## Verify M2 is healthy

```bash
cd ~/sensei-client
source venv/bin/activate
set -a; source .env; set +a

python -c "import sensei_client; print(sensei_client.__version__)"
# → 0.3.0

curl -f "$SENSEI_API_URL/adapters"
# → 200 OK, returns list of adapters registered on M3

python tests/e2e_smoke.py
# → 4/4 scenarios passed
```

## Common issues

- **M3 unreachable from M2.** Check firewall / VPN / subnet. `curl -v $SENSEI_API_URL/adapters` tells you whether it's a DNS issue, connection refused, or timeout. sensei_client can't bypass a network gap.
- **`pip install git+https://...` prompts for credentials.** The repo is public — no auth needed. If you see a prompt, you're hitting a stale git credential helper. Clear it: `git credential-cache exit`.
- **Smoke test scenarios 2/3 fail but 1/4 pass.** Likely M3 rejected the adapter registration. Check M3's logs. Possibly a contract version mismatch (M3 running older SENSEI service).
- **Smoke test always shows "Never used" on the token metadata.** Ignore — unrelated to M2 wiring. That's a GitHub UI lag, not a sensei_client signal.

## After bootstrap

M2 is now a standard sensei_client consumer. Use it like any other dependency. The `~/sensei-client` checkout is only there to give you the adapter specs, smoke test, and contract reference — it's not required to be present for projects to consume sensei_client, but it's useful to have around.
