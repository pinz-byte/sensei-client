# M3 — SENSEI Service Host (and optional consumer)

**Role:** Runs the SENSEI service on port 8000. Is authoritative for contract enforcement, adapter registration state, event logging, and calibration. May also consume `sensei_client` if M3 hosts any ventures.

**Network:** Accepts connections from M1, M2, and any venture runtime. Connects outbound to Anthropic API (for Advisor calls, if plugin-path adapters use it).

## Prerequisite — SENSEI service must already be running

Per the prior state (176 tests passing), M3 is already hosting the SENSEI service. This setup pack is for the **client side** of M3 — the part that consumes `sensei_client` if M3 also runs a venture.

If the SENSEI service is not yet running on M3, that's a separate setup (the `sensei-service` repo, which is not yet extracted — contract and advisories currently live in the `sensei-client` repo as archives, per `HANDOFF.md`).

Assumption below: SENSEI service is up at `http://localhost:8000` on M3.

## Bootstrap

```bash
# Either clone the public repo or operate inside your existing M3 sensei-service checkout
git clone https://github.com/pinz-byte/sensei-client.git ~/sensei-client
cd ~/sensei-client

# Bootstrap M3 role — no --m3-url needed, defaults to localhost:8000
./setup/bootstrap.sh M3
```

What it does on M3 specifically:

1. Verifies Python ≥ 3.9
2. Creates `venv/`
3. Installs `sensei_client` from the v0.3.2 git tag
4. Writes `.env` with `SENSEI_STATION=M3` and `SENSEI_API_URL=http://localhost:8000`
5. Runs the smoke test against the local SENSEI service

## M3-specific tasks after bootstrap

### Register a venture adapter (JSON path)

If M3 hosts a venture (HERALD / AVT Extractor / Subastop), that venture registers its adapter with the local SENSEI at startup. The bootstrap leaves you with the HERALD spec available at `sensei_adapters/herald/herald.v1.json`. To register AVT Extractor:

```bash
source venv/bin/activate
set -a; source .env; set +a

python -c "
from sensei_client import SenseiConfig, SenseiClient
c = SenseiConfig.from_spec_file('sensei_adapters/avt_extractor/avt_extractor.v1.json',
                                 api_url='$SENSEI_API_URL')
with SenseiClient(c) as cli:
    cli.register_from_config()
    print(f'registered: {c.adapter_id}')
"
```

Registrations are in-memory. SENSEI service restart drops them. Ventures own the re-registration lifecycle (contract v0.3 §3.5, Invariant I14).

### Contract and advisories — migration to `sensei-service`

Long-term per 3-repo topology, `contract/` and `advisories/` live in the `sensei-service` repo on M3, not here in `sensei-client`. Until that migration happens (see `HANDOFF.md` §3), M3 reads the archive copies from this clone. No action needed on M3's side — just note the eventual destination.

### Verify M3 is healthy

```bash
source venv/bin/activate
set -a; source .env; set +a

# Local service reachable
curl -f http://localhost:8000/adapters
# → 200 OK

# sensei_client can round-trip
python tests/e2e_smoke.py
# → 4/4 scenarios passed

# Registered adapters
curl -s http://localhost:8000/adapters | python -m json.tool
# → list of registered adapters (herald.v1, possibly others)
```

## Common issues

- **Smoke test scenario 4 (fail-open) fails on M3.** Scenario 4 points at `127.0.0.1:1` intentionally. If your M3 has a service listening on port 1 (unlikely), or if the local SENSEI accepts connections on ANY port, the fail-open path won't trigger. Not a blocker.
- **Adapter not persisting across restart.** By design — v0.3 §3.5. Ventures MUST re-register on startup. Treat `404` from `/decide` as a signal to re-register and retry (sensei_client's guard does this automatically).
- **Advisor calls failing.** M3 needs outbound reach to Anthropic API and the `anthropic` Python package (`pip install "sensei_client[advisor]"` picks it up). Check `ANTHROPIC_API_KEY` is set in the environment where ventures run.
- **Contract version mismatch.** If a venture registers with `contract_version: "0.3"` but M3 service rejects with 400, M3 may still be running a pre-v0.3 build. Confirm the SENSEI service on M3 is at v0.3 before pointing adopters at it.

## Separation of concerns

`sensei_client` installed on M3 is the *consumer* library — same code other stations use. It is structurally separate from the SENSEI server code running on M3. The server doesn't depend on the client; the client doesn't configure the server. They meet only on the wire (HTTP contract v0.3).
